# Work Skill 分析 Prompt

## 任務

你將收到 **{name}** 的原材料（文件、訊息、郵件等）。
從中提取他的工作能力與方法，用於建構 Work Skill。

**原則：只提取工作相關內容，忽略閒聊。不要推斷，有依據才寫，沒有就標注「原材料不足」。**

---

## 通用提取維度（所有職位適用）

### 1. 負責範圍

從原材料中識別：
- 他負責的系統/模組/業務線/產品
- 他維護的文件（介面文件、wiki、runbook...）
- 他的職責邊界（哪些是他的，哪些不是）
- 他頻繁提到的專案代號、業務術語

```
輸出格式：
負責領域：[描述]
核心系統：[列表]
維護文件：[列表]
邊界：[他管什麼/不管什麼]
```

### 2. 工作流程

從任務描述、會議紀要中提取：
- 接到任務的處理步驟
- 寫方案/文件的結構習慣
- 如何做進度管理和 deadline 處理
- 如何處理異常/緊急情況

```
輸出格式：
接任務：[步驟]
寫方案：[結構描述]
異常處理：[流程]
```

### 3. 輸出格式偏好

- 用表格/列表/流程圖/純文字
- 結論前置還是娓娓道來
- 文件詳細程度（極簡/適中/詳盡）
- 回覆/郵件風格

```
輸出格式：
文件風格：[描述]
詳細程度：[極簡/適中/詳盡]
```

### 4. 經驗知識庫

他明確表達的經驗判斷、踩過的坑、技術觀點（直接引用原話）：

```
- "[原話或總結]"
- "[原話或總結]"
```

## 職位專項提取

根據 {name} 的職位，重點提取對應維度：

---

### 🖥️ 後端工程師 / 服務端工程師

**技術規範**：
- 技術棧（語言、框架、中介軟體）
- 命名規範（介面路徑風格、變數/函式命名）
- 介面設計（回傳結構、錯誤碼、分頁、冪等）
- 資料庫操作偏好（ORM vs 原生 SQL，交易邊界）
- 例外處理風格

**Code Review 重點**：
- 他反覆提到的 CR 問題（N+1、交易、並行安全...）
- 他的 CR 評論風格（直接/委婉，[block]/[suggest] 分級...）

**部署與維運**：
- 他關注的監控指標
- 線上問題排查步驟
- 變更發佈流程

---

### 🌐 前端工程師

**技術規範**：
- 技術棧（框架、狀態管理、樣式方案）
- 元件拆分原則（什麼時候拆，什麼時候不拆）
- 效能關注點（首屏、懶載入、bundle 大小...）
- 介面呼叫和錯誤處理方式

**工程實踐**：
- 程式碼規範工具（ESLint 規則、Prettier 設定偏好）
- 測試覆蓋要求（單測/整合測試態度）
- CR 重點（無障礙/響應式/相容性關注度）

---

### 🤖 演算法工程師 / ML 工程師

**研究與實驗**：
- 問題定義方式（如何拆解 ML 問題）
- 實驗設計習慣（基線選擇、ablation 設計）
- 指標定義偏好（離線指標 vs 線上指標的態度）
- 他常用的模型/方法論

**工程落地**：
- 訓練框架偏好
- 模型部署流程
- 資料處理規範

**文件與結論**：
- 實驗報告的寫法（重結論/重過程）
- 他引用的 paper 或方法論

---

### 📱 產品經理 / 技術產品經理

**需求處理**：
- PRD 結構和詳細程度
- 使用者故事/需求邊界的定義方式
- 如何與研發對齊（審查方式、修改流程）

**決策框架**：
- 優先級排序方法（RICE/MoSCoW/自訂）
- 數據驅動 vs 直覺的比例
- 如何處理需求衝突

**產出物**：
- 他交付的文件類型（PRD/MRD/原型/競品分析）
- 原型工具偏好
- 數據埋點的參與程度

---

### 🎨 設計師

**設計規範**：
- 使用的設計系統/元件庫
- 標註方式和交付規範
- 對 pixel-perfect 的要求程度

**工作流程**：
- 從需求到方案的步驟
- 走查/驗收的方式
- 如何處理開發端的還原度問題

---

### 📊 資料分析師

**分析方法**：
- 常用分析框架（漏斗/同期群/A/B 測試...）
- SQL 風格（簡潔/註解詳盡）
- 資料視覺化偏好（圖表類型選擇）

**報告風格**：
- 結論 vs 數據的比例
- 對「數據說話」的執行程度
- 如何處理數據異常/口徑爭議

---

## 輸出要求

- 語言：正體中文
- 沒有資訊的維度：標注 `（原材料不足，建議追加相關文件）`
- 有原文依據的結論：加引號標注原話
- 輸出結果直接用於產生 work.md，要求具體可執行，不要寫「可能」「傾向於」這類模糊表述
